# Precision Testing Suite

## Overview
This folder contains the automated test suite designed to determine the minimum floating-point-to-integer scaling factor (`HSS_SCALE`) required for the LWE-based face authentication system.

Because LWE lattice cryptography requires integer operations, the continuous 128-dimensional floating-point face vectors (generated by neural networks) must be quantized. 
This test determines how much precision we can discard (to optimize cryptographic modulus $P$ and $Q$ size) without introducing biometric inaccuracies.

## Architecture of the Test
1. **Dynamic Scaling:** The script compiles the core cryptography engine repeatedly, dynamically injecting different `HSS_SCALE` constants (from $2^{16}$ down to $2^7$).
2. **Distribution Balance:** For each precision level, it evaluates **50,000 pairs** of faces.
   - **~10% (5,000 pairs) are matched faces (Same Identity):** Tests for **False Reject Rate (FRR)** degradation. If precision is too low, valid matches fail to reach the threshold.
   - **~90% (45,000 pairs) are mismatched faces (Different Identities):** Tests for **False Accept Rate (FAR)** vulnerabilities. If precision is too low, the system might round up a near-match imposter, incorrectly authenticating them.
3. **Dual-Domain Execution:** Every single pair is evaluated twice:
   - **Plaintext Domain:** Standard floating-point Cosine Similarity computation.
   - **Cryptographic Domain:** Quantized, Homomorphically encoded integer inner-product matching.
   If the binary outcome (Accept/Reject) of the Cryptographic domain deviates from the Plaintext domain, an error is recorded.

## Running the Test
```bash
chmod +x run_test.sh
./run_test.sh
```

## Understanding the Output Metrics
The test suite now outputs two critical mathematical metrics for each scale, detailing the exact quantization loss incurred when moving from floating-point to LWE integer space:

1. **Mean Absolute Error (Cosine Dist):**
   - Measures the absolute numerical difference between the Plaintext Cosine Similarity (e.g., `0.912345`) and the Cryptographic reconstructed similarity (e.g., `0.912343`).
   - *Example (`0.000002`):* Indicates the integer calculation is accurate to the 5th decimal place. To trigger a False Accept (FAR) at this scale, an imposter's true similarity would have to be exactly `0.899998`, resting precisely on the razor's edge of the `0.90` threshold.

2. **Average Relative Error Percentage:**
   - Measures the quantization error relative to the original similarity score.
   - *Example (`0.0002%`):* Indicates the cryptographic system maintains **99.9998% fidelity** to the original continuous neural network vector math. 

As the `HSS_SCALE` decreases, these errors will grow. When the Absolute Error exceeds the natural variance of faces near the threshold, FAR and FRR mismatches will begin to trigger.

## Threshold Dependency Notice
**Important:** The safety limits discovered by this test are strictly tied to the current authentication threshold (defined by `FACE_TAU_NUM` / `FACE_TAU_DEN` in `face_api.h`, default `0.9`). 

If you lower the similarity threshold (e.g., to `0.85`), the density of face vectors residing near the threshold boundary will increase dramatically. This will cause precision errors to manifest at much higher scale factors. **Always re-run this suite if you adjust the biometric threshold.**
